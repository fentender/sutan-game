# 格式规则文件系统设计

## 背景

合并器（merger.py）通过 schema 规则文件感知每个字段的类型和合并策略，替代原有的纯结构推断。每类 JSON 文件对应一个 schema，合并时参考规则决定具体合并行为。

---

## 一、规则文件设计

### 存放位置

`schemas/` 目录，共 **26 个**规则文件：

- 根文件 18 个：`cards.schema.json`、`tag.schema.json`、`variable.schema.json` 等
- 子目录 8 个：`rite.schema.json`（覆盖 rite/ 目录所有文件）、`event.schema.json` 等

### 规则文件结构

```json
{
  "_meta": {
    "schema_version": 1,
    "file_type": "dictionary|entity|config",
    "description": "卡牌定义表",
    "source": "cards.json"
  },
  "_entry": {
    "id":       { "type": "int",    "merge": "replace" },
    "name":     { "type": "string", "merge": "replace" },
    "resource": { "type": ["string", "array<string>"], "merge": "coerce" },
    "tag": {
      "type": "object",
      "merge": "merge",
      "dynamic_keys": true,
      "dynamic_value": { "type": "int", "merge": "replace" }
    },
    "settlement_prior": {
      "type": "array<object>",
      "merge": "smart_match",
      "match_strategy": "rite",
      "element": {
        "guid":         { "type": "string", "merge": "replace" },
        "condition":    { "type": "object", "dynamic_keys": true, "merge": "replace" },
        "result":       { "type": "object", "dynamic_keys": true, "merge": "merge" }
      }
    }
  }
}
```

**顶层 key 说明**：

- `_meta` — 文件元信息（schema_version, file_type, description, source）
- `_entry` — dictionary 类型文件，描述每个条目（value）的字段
- `_fields` — entity/config 类型文件，描述顶层字段
- `_dynamic_value` — 整个文件是动态 key 的情况（如 sfx_npc_role_dub.json）

### 字段定义属性

| 属性 | 类型 | 说明 |
| --- | --- | --- |
| `type` | `string \| string[]` | 字段类型：`int`, `float`, `string`, `bool`, `object`, `array`, `array<int>`, `array<string>`, `array<object>`, `null` |
| `merge` | `string` | 合并策略：`replace`, `merge`, `append`, `smart_match`, `coerce` |
| `dynamic_keys` | `bool` | object 类型是否是动态 key（如 tag、condition、action） |
| `dynamic_value` | `field_def` | 动态 key 的兜底值规则（未被 fields 匹配时使用），支持 `merge_by_type` |
| `merge_by_type` | `{type: strategy}` | 按实际值类型分派合并策略（仅用于 dynamic_value） |
| `fields` | `{name: field_def}` | 固定 key 或已知子结构的字段规则（优先于 dynamic_value） |
| `element` | `{name: field_def}` | array\<object\> 的元素字段规则 |
| `match_strategy` | `string` | smart_match 时的匹配策略：`rite` 或 `event` |

### 合并策略含义

| 策略 | 行为 |
| --- | --- |
| `replace` | 直接替换（int/string/bool 等标量） |
| `merge` | 递归深度合并（object） |
| `append` | 数组追加去重 |
| `smart_match` | 智能匹配后逐元素合并（settlement 类数组） |
| `coerce` | 类型转换合并：标量+数组 → 数组 |

### dynamic_keys + fields 共存

动态 key 对象（如 `action`）支持 `dynamic_keys: true` 和 `fields` 同时存在：

- `fields` 记录已知固定结构的子 key（如 `prompt`、`confirm`、`option`、`delay`）
- `dynamic_value` 作为未知 key 的兜底规则
- 查找优先级：`fields` > `dynamic_value`

示例——`action` 字段的 schema：

```json
{
  "type": "object",
  "merge": "merge",
  "dynamic_keys": true,
  "dynamic_value": {
    "type": ["int", "object"],
    "merge": "replace",
    "merge_by_type": { "object": "merge" }
  },
  "fields": {
    "prompt": {
      "type": "object", "merge": "merge",
      "fields": { "id": {"type":"string","merge":"replace"}, "text": {"type":"string","merge":"replace"}, "icon": {"type":["string","array<string>"],"merge":"coerce"} }
    },
    "confirm": {
      "type": "object", "merge": "merge",
      "fields": { "id": {"type":"string","merge":"replace"}, "text": {"type":"string","merge":"replace"}, "confirm_text": {"type":"string","merge":"replace"}, "cancel_text": {"type":"string","merge":"replace"} }
    },
    "delay": {
      "type": "object", "merge": "merge",
      "fields": { "id": {"type":"int","merge":"replace"}, "round": {"type":"int","merge":"replace"}, "event_on": {"type":"int","merge":"replace"} }
    }
  }
}
```

这样 `action.prompt` 使用精确的字段规则合并，而 `action.rite`（int）、`action.case:op3`（object）等使用 `dynamic_value` 兜底。

### 按类型分派（merge_by_type）

`dynamic_value` 支持为不同值类型指定不同合并策略。例如 `action` 的动态 key 值可能是 int（如 `rite: 5006075`）或 object（如 `case:op3: {...}`），需要不同策略：

```json
"dynamic_value": {
  "type": ["int", "object"],
  "merge": "replace",
  "merge_by_type": { "object": "merge" }
}
```

查找顺序：`merge_by_type[实际值类型]` > `merge`（默认）

效果：`action.rite`（int）→ replace 直接替换；`action.case:op3`（object）→ merge 递归合并不丢失字段。

无 schema 时：dict + dict 默认 merge，其他默认 replace。

### 合并策略的配置层级

schema 定义每个字段的默认合并策略。用户可在 GUI 工具中对特定字段覆盖策略：

- GUI 提供字段级别的策略配置界面
- 配置保存在 `user_config.json` 中（如 `"schema_overrides": {"rite.schema.json": {"condition": "merge"}}`）
- 合并时优先级：用户配置 > schema 默认值

## 二、自动生成脚本

**文件**：`scripts/generate_schemas.py`

### 功能

- 读取游戏本体 `config/` 目录下所有文件
- 复用 `json_parser.py` 的 JSON 清理逻辑
- 对每个根文件/子目录采样分析（默认采样 80 个文件），推断字段类型和合并策略
- 输出 26 个 `.schema.json` 到 `schemas/`

### 类型推断规则

- 仅见 int → `"int"`
- 仅见 string → `"string"`
- int+float 混合 → `"float"`
- string+array 混合 → `["string", "array<string>"]`
- object 且子 key > 30 种 → `dynamic_keys: true`
- 已知 DSL 字段（condition/result/action/tag/cards_slot 等）→ 强制 `dynamic_keys: true`

### 合并策略推断规则

- 标量（int/float/string/bool） → `"replace"`
- 固定/动态 object → `"merge"`
- 联合类型含 object → `"merge"`（非 object 联合类型 → `"coerce"`）
- array\<object\> 且元素含 guid/condition → `"smart_match"`
- 已知智能合并字段（settlement/settlement_prior/settlement_extre/waiting_round_end_action）→ `"smart_match"`
- 普通 array → `"append"`（默认追加去重）

### 动态 key 对象的处理

生成脚本对 `dynamic_keys: true` 的对象：

1. 遍历所有子 key，收集全部类型 → 生成 `dynamic_value`（完整兜底）
2. 子 key 中值类型仅为 object 且有固定子结构的 → 提取到 `fields`
3. `fields` 和 `dynamic_value` 并存，查找时先匹配 fields 再兜底

## 三、核心模块

### schema_loader.py

`src/core/schema_loader.py`，负责 schema 加载和字段查找。

核心函数：

- `load_schemas(schema_dir)` — 加载所有 .schema.json，返回 `{pattern: schema}` 映射
- `resolve_schema(rel_path, schemas)` — 根据文件路径匹配 schema（精确匹配根文件 `cards.json`，目录匹配子目录文件 `rite/` → `rite.schema.json`）
- `get_field_def(schema, field_path)` — 在 schema 树中导航查找字段定义
- `check_type_match(schema_type, actual_value)` — 检查值类型是否匹配 schema

`get_field_def` 导航优先级：

1. `_entry` / `_fields` — 顶层入口
2. 直接 key 匹配
3. `fields` — 固定子字段
4. `element` — 数组元素字段
5. `dynamic_value` — 动态 key 兜底

### merger.py

`src/core/merger.py`，schema 驱动的合并逻辑。

`deep_merge(base, override, schema=None, field_path=None)`:

1. 根据 `field_path` 从 schema 查找字段定义
2. 由 `merge` 策略决定行为（replace/merge/append/smart_match/coerce）
3. 安全回退：schema 说 replace 但两边都是 dict 时用 merge
4. 类型校验：不匹配时记录到 `merge_warnings`，不阻断
5. 未知 key 检测：非 `dynamic_keys` 层的未知 key 记录警告

`merge_file(base_data, mod_data_list, rel_path, schema)`:

- dictionary 类型：按顶层 key 逐个合并，field_path 从 `_entry` 开始
- entity/config 类型：整体合并，field_path 从 `_fields` 开始

`merge_all_files(game_config_path, mod_configs, output_path, schema_dir)`:

- 启动时加载 schemas（从 `schemas/` 目录）
- 为每个文件匹配 schema，传递给 merge_file

## 五、验证方式

1. 运行 `python scripts/generate_schemas.py` 检查 26 个 schema 文件完整生成
2. 用现有 mod 文件测试合并，验证结果正确
3. 特别验证：
   - cards.json 的 `resource` 字段（string|array 联合类型 coerce）
   - rite/*.json 的 settlement 数组（smart_match + guid 匹配）
   - tag.json 的动态 key 合并（新 tag 追加，已有 tag 覆盖）
   - action 中 case:op3 合并不丢失 rite 等未修改字段
   - prompt/confirm/option 等已知子结构的递归合并
   - 未知 key 警告是否正确触发
   - 类型不匹配警告是否正确触发
